## 0. Tesi dell'audit (confermata)

Il consolidamento del grafo (#24/#25/#27/#28/#29, 13 tool MCP) è tecnicamente solido, ma
**non è il valore unico di RTK e sovrappone gitnexus**. L'audit conferma numericamente
l'insight di fine sessione precedente:

- **Il valore unico (efficienza token) è sotto-costruito**: la superficie filtri copre solo
  **8 famiglie di comando** wired nel dispatch.
- **Il grafo consuma peso sproporzionato**: `rtk-index` (2057 LOC) + `rtk-mcp` (1145 LOC) +
  5 grammatiche tree-sitter + sqlite bundled = ~metà del binario (8.8 MB), per una feature che
  duplica gitnexus e ha gap di precisione noti (callee_file_path 0%, "no upstream" residui).

→ **Il ROI è nei filtri/pack/DLP, non nel grafo.** Il piano sotto è ordinato di conseguenza.

## 2. Findings per crate

### rtk-filters (1900 LOC, 14 file) — 🟢 il cuore del valore, ben fatto ma stretto
- **Copertura reale wired in `dispatch.rs`:** git (diff/status/log/show/branch), cargo (test/build/check),
  npm, pytest, ls, go test, docker, gradle → **8 famiglie**.
- Ogni filtro ha `#[cfg(test)]` (**13/13**) con pattern snapshot/savings. Ottimo template.
- **Gap:** i `.claude/rules` (dev-root) descrivono un RTK idealizzato (pnpm/vitest/tsc/eslint/ruff/
  mypy/pip/dotnet/kubectl/psql/gh…) che **in questo repo non esiste**. Nessuna dir js/python/dotnet/
  cloud/ruby, nessun filtro TOML. La superficie token è molto più sottile della narrativa.
- `npm_filter` prod unwrap = init regex in `LazyLock` → pattern accettato RTK, non un rischio.

### rtk-db (3605 LOC, 9 file) — 🟢 DLP è un vero differenziatore
- **DLP (`dlp.rs`) robusto e ampliato:** ora copre `sk-proj`/`sk-ant`/`sk-`/`sk_live|test`/`ghp_`/
  `xox*`/`AIza`/`AKIA`, PEM blocks, JWT `eyJ…`, credenziali in URL. Il gap storico `sk-` (issue #15) è
  **chiuso**. Escaping CDATA-safe in pack. Punto di forza sotto-comunicato.
- `dlp.rs` prod unwrap: 3 `Regex::new().unwrap()` (init, ok) + `caps.get(0).unwrap()` (gruppo 0 sempre
  presente, safe).
- tracking/config/session/artifact/pricing/think: 7/9 file testati. Flake Windows config (issue) risolto
  in #26.

### rtk-pack (687 LOC, 3 file) — 🟢 solido
- `pack_directory` integra **DLP su ogni file** (`redact_with_source`) + strip commenti + skeleton mode.
  Sicurezza-by-default forte. `strip_content` = savings diretti.

### rtk-cli (5415 LOC, 18 file) — 🟡 glue sotto-testato + papercut robustezza
- **`setup.rs`: 6 `unwrap()` su manipolazione JSON di settings.json** (`as_object_mut().unwrap()`,
  `get_mut("hooks").unwrap()`). Se l'utente ha un settings.json non standard → panic in `rtk setup`.
  Viola la regola no-panic-in-prod. **User-facing.**
- **`dotnet.rs:10`: `unwrap_or_else(|_| panic!("Failed to execute dotnet command"))`** → panic se
  dotnet non installato. Viola la regola fallback-to-raw.
- **Test: solo 5/18 file cli** hanno `#[cfg(test)]`. `dispatch.rs` e `rewrite.rs` (i veri entry-point)
  poco coperti a livello integrazione.

### rtk-index (2057 LOC, 5 file) — 🟡 completo ma peso/ROI sfavorevole
- Grafo AST via tree-sitter (5 grammatiche), FTS, embeddings **già feature-gated** (`default = []`, no ONNX).
- Residui noti (dal piano): `callee_file_path` popolato 0% → archi name-based; `impact compute_levels`
  "no upstream". Migliorabili ma **duplicano gitnexus**: non è qui il ROI.

### rtk-mcp (1145 LOC, 1 file) — 🟢 13 tool, ergonomia buona
- Superficie MCP completa (symbols/refs/impact/detect-changes/rename/flow…). Da **validare sul campo**
  come rimpiazzo gitnexus (§2 del piano di sessione), non da estendere.

## 3. Piano prioritizzato per ROI

Ordinato per ritorno/(effort), coerente con la tesi "il valore è token, non grafo".

### 🥇 P1 — Ampliare la superficie filtri (ALTO ROI, effort BASSO-MEDIO)
Il moltiplicatore di valore di RTK. Template + harness (snapshot + savings test) **già esistono**;
aggiungere un filtro = copiare pattern, non inventare infrastruttura.
- Candidati ad alta frequenza agentica, in ordine: **`tsc`/eslint**, **`ruff`/`mypy`/`pip`**,
  **`pnpm`/`vitest`**, **`gh` (pr/run view)**, **`kubectl`/`docker ps`**.
- Per ciascuno: fixture reale → `assert_snapshot!` → assert savings ≥60%. 1 filtro ≈ 0.5–1 gg.
- **Deliverable:** +4–6 famiglie di comando ⇒ raddoppia la superficie token con rischio minimo.

### 🥈 P2 — Quick-win robustezza (MEDIO ROI, effort BASSO — <1 gg)
Papercut user-facing che violano regole RTK dichiarate.
1. `setup.rs`: sostituire i 6 `unwrap()` JSON con `.context()?` + fallback (non corrompere settings.json).
2. `dotnet.rs:10`: rimuovere `panic!`, passthrough al comando raw se dotnet manca.
3. `anyhow` bump per chiudere RUSTSEC-2026-0190 (warning, low).

### 🥉 P3 — Test integrazione dispatch/rewrite (MEDIO ROI, effort MEDIO)
`dispatch.rs`/`rewrite.rs` sono gli entry-point reali ma sotto-testati (cli 5/18).
- Test end-to-end: comando → filtro giusto, passthrough su comando ignoto, exit-code propagato.
- Alza Test 7.0→8+ e blinda il percorso critico.

### 4 — Validazione grafo sul campo (secondaria, veloce — da §2 del piano)
Nel worktree `cyclelab-terminal-rtktest` (2111 sym), provare i 13 tool MCP vs gitnexus offline.
**Non estendere il grafo** finché la validazione non mostra un buco reale. Se emerge, l'unico intervento
giustificato è `callee_file_path` (migliora impact+detect+flow insieme).

### 5 — HTML graph viewer (OPZIONALE, valore visuale, clean-room)
`rtk graph export --format html` self-contained dal nostro DB (cytoscape via CDN). Solo se l'utente lo
vuole; **non** copiare la web UI gitnexus (PolyForm Noncommercial). Basso in classifica: è grafo.

### Non fare (esplicito)
- ❌ Investire altro sul grafo/precisione archi come priorità: duplica gitnexus, peso binario alto, ROI basso.
- ❌ Portare codice gitnexus (TS→Rust = opera derivata, viola PolyForm, contamina Apache-2.0). Solo clean-room.
- ❌ Ridurre il binario disabilitando tree-sitter: romperebbe il grafo; se mai, gate dell'intero index
  dietro feature — ma è lavoro senza ritorno-utente diretto.
